# snapeval: Align to agentskills.io Evaluation Spec

**Date:** 2026-03-19
**Status:** Draft

## Goal

Refactor snapeval to implement the [agentskills.io evaluation spec](https://agentskills.io/skill-creation/evaluating-skills) exactly. snapeval becomes a harness-agnostic eval runner — users bring their own harness (Claude Code, Copilot CLI, Copilot SDK, etc.), snapeval orchestrates the eval workflow and produces the spec's artifacts.

No backward compatibility. Everything not in the spec gets removed.

## CLI Commands

Three commands mapping to the spec's workflow phases:

### `snapeval init <skill-dir>`

Generates `evals/evals.json` from SKILL.md using an LLM. Per the spec, `init` generates prompts and expected outputs only — **not assertions**. Assertions are added by the user after seeing first outputs from `eval`.

**Flags:** `--harness`, `--inference`, `--verbose`

### `snapeval eval <skill-dir>`

The core automated pipeline. For each eval case in `evals.json`:

1. **Run with skill** — invoke the harness with SKILL.md loaded, save outputs + timing
2. **Run without skill** — invoke the harness without SKILL.md, save outputs + timing
3. **Grade assertions** — if assertions exist, grade each against outputs. Uses verification scripts for mechanical checks (file exists, valid JSON, row counts) and the inference LLM for semantic checks. Produces grading.json per run.
4. **Aggregate** — compute benchmark.json with `{with_skill, without_skill, delta}` stats

Creates a new `iteration-N/` directory in the workspace for each invocation.

**Flags:** `--harness`, `--inference`, `--workspace`, `--runs <N>` (repeat each eval N times for statistical significance), `--verbose`

**Comparing skill versions:** When `--old-skill <path>` is provided, the baseline run uses the old skill instead of running without a skill. Outputs go to `old_skill/` instead of `without_skill/`. The spec recommends blind comparison for version comparisons: both outputs are presented to the grading LLM without revealing which came from which version.

### `snapeval review <skill-dir>`

Runs `eval` then generates an HTML report and opens it in the browser. Creates a template `feedback.json` with empty strings for each eval slug so the user can fill in feedback directly.

**Flags:** same as `eval` plus `--no-open`

## evals.json Format

Per the agentskills.io spec exactly:

```json
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "I have a CSV of monthly sales data in data/sales_2025.csv. Can you find the top 3 months by revenue and make a bar chart?",
      "expected_output": "A bar chart image showing the top 3 months by revenue, with labeled axes and values.",
      "files": ["evals/files/sales_2025.csv"],
      "assertions": [
        "The output includes a bar chart image file",
        "The chart shows exactly 3 months",
        "Both axes are labeled",
        "script:verify-chart.sh"
      ]
    }
  ]
}
```

- `skill_name`: required
- `evals[].id`: required, numeric
- `evals[].prompt`: required
- `evals[].expected_output`: required, human-readable description of success
- `evals[].files`: optional, input files the skill needs
- `evals[].slug`: optional, short kebab-case label for directory naming (e.g. `top-months-chart`). Generated by the inference LLM during `init` from a 2-4 word summary of the prompt. Falls back to `eval-{id}` if not present.
- `evals[].assertions`: optional, added after first `eval` run. Each is a string — either a natural language assertion (graded by LLM) or prefixed with `script:` to invoke a verification script.

`init` generates evals **without** assertions. The user adds assertions after reviewing first outputs.

## Harness Abstraction

The single extension point. A harness implements two operations:

1. **Run with skill** — start a fresh agent session with SKILL.md loaded, execute the prompt, capture outputs
2. **Run without skill** — start a fresh agent session without SKILL.md, execute the same prompt, capture outputs

**Session isolation is required.** Each run must start with a clean context — no leftover state from previous runs or from the skill development process. This ensures the agent follows only what the SKILL.md tells it. Harness implementations must guarantee this (e.g., separate subprocess per run).

Each run returns:
- `raw`: the agent's text output
- `transcript`: the full execution log (step-by-step agent actions), if available from the harness
- `files`: any files produced (paths relative to outputs/)
- `total_tokens`: token count for the session
- `duration_ms`: wall-clock time

Input files from the eval case are copied into the run's working directory before execution. The harness receives absolute paths to the copied files.

Built-in harness: `copilot-cli` (uses `gh copilot`).

Selected via `--harness` flag or `snapeval.config.json`. The interface is public so users can implement custom harnesses.

## Inference Layer

Separate from the harness. Used for:
- Generating evals.json from SKILL.md (`init`)
- Grading semantic assertions against outputs (`eval`)

Built-in: `copilot` (free via Copilot CLI), `github-models` (free via GITHUB_TOKEN). Auto-resolution with fallback chaining.

## Grading

Two grading mechanisms, per the spec:

### LLM-based grading (default)

For natural language assertions. The inference LLM evaluates each assertion against the run's outputs and returns PASS/FAIL with evidence quoting the output. Requires concrete evidence for a PASS — no benefit of the doubt.

### Script-based grading

For mechanical checks. Assertions prefixed with `script:` invoke a verification script from `evals/scripts/`. The script receives the outputs directory as its argument and exits 0 (pass) or 1 (fail), writing evidence to stdout. Scripts are more reliable than LLM judgment for checks like "valid JSON", "file exists", "correct row count."

### Blind comparison (version comparison mode)

When using `--old-skill`, both outputs are presented to the LLM grader without labels identifying which came from which version. The grader scores holistic qualities — organization, formatting, usability — on its own rubric. This complements assertion grading for catching quality differences that assertions don't cover.

## Workspace Structure

Exactly per the agentskills.io spec. Workspace is a sibling directory `{skill-name}-workspace/` by default, configurable via `--workspace`.

```
skill-dir/
├── SKILL.md
└── evals/
    ├── evals.json
    └── scripts/          # Verification scripts for script: assertions
        └── verify-chart.sh

skill-dir-workspace/
└── iteration-1/
    ├── eval-{slug}/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── transcript.log   # Full execution log (if available)
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/        # or old_skill/ when using --old-skill
    │       ├── outputs/
    │       ├── transcript.log
    │       ├── timing.json
    │       └── grading.json
    ├── benchmark.json
    └── feedback.json
```

Eval directory slug is derived from the eval id and a short LLM-generated label during `init` (stored as an optional `slug` field in evals.json), falling back to `eval-{id}` if not present.

## Artifact Formats

All formats match the agentskills.io spec exactly.

### timing.json

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

### grading.json

```json
{
  "assertion_results": [
    {
      "text": "The output includes a bar chart image file",
      "passed": true,
      "evidence": "Found chart.png (45KB) in outputs directory"
    },
    {
      "text": "Both axes are labeled",
      "passed": false,
      "evidence": "Y-axis is labeled 'Revenue ($)' but X-axis has no label"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

### benchmark.json

```json
{
  "run_summary": {
    "with_skill": {
      "pass_rate": { "mean": 0.83, "stddev": 0.06 },
      "time_seconds": { "mean": 45.0, "stddev": 12.0 },
      "tokens": { "mean": 3800, "stddev": 400 }
    },
    "without_skill": {
      "pass_rate": { "mean": 0.33, "stddev": 0.10 },
      "time_seconds": { "mean": 32.0, "stddev": 8.0 },
      "tokens": { "mean": 2100, "stddev": 300 }
    },
    "delta": {
      "pass_rate": 0.50,
      "time_seconds": 13.0,
      "tokens": 1700
    }
  }
}
```

`stddev` has two semantics depending on `--runs`:
- **N > 1:** stddev measures within-eval variance (flakiness of a single eval across repeated runs).
- **N = 1:** stddev measures across-eval variance (difficulty spread across different eval cases).

The HTML report labels these differently so users understand what the number means.

### feedback.json

Template generated by `review` with empty strings for each eval. User fills in feedback by hand:

```json
{
  "eval-top-months-chart": "The chart is missing axis labels and the months are in alphabetical order instead of chronological.",
  "eval-clean-missing-emails": ""
}
```

Empty string = output looked fine. Only non-empty entries represent actionable feedback.

## Iteration Workflow

The spec defines a concrete iteration loop. snapeval supports it through its artifacts:

1. `snapeval init` → generates evals.json (user reviews/edits)
2. `snapeval eval` → first run, produces outputs (user reviews, adds assertions to evals.json)
3. `snapeval eval` → second run with assertions, produces grading + benchmark
4. `snapeval review` → HTML report + template feedback.json (user fills in feedback)
5. User feeds failed assertions + feedback.json + transcripts + current SKILL.md to an LLM → proposes SKILL.md improvements
6. User applies changes, runs `snapeval eval` again in a new iteration
7. Repeat until satisfied

snapeval does not automate step 5 (the LLM-assisted skill improvement) — that's the user's workflow. snapeval produces the artifacts that feed into it: grading.json (failed assertions), feedback.json (human feedback), transcript.log (execution traces), and benchmark.json (progress tracking across iterations).

## Pattern Analysis

The `review` HTML report highlights patterns from the spec's "Analyzing patterns" section:

- **Always-pass assertions** — pass in both with_skill and without_skill. Flagged as "not differentiating."
- **Always-fail assertions** — fail in both configurations. Flagged as "possibly broken."
- **Differentiating assertions** — pass with_skill, fail without. These show where the skill adds value.
- **High stddev** — inconsistent results suggesting flaky evals or ambiguous skill instructions.
- **Time/token outliers** — evals that cost significantly more than average.

## What Gets Removed

- Snapshot system (`engine/snapshot.ts`, `commands/approve.ts`, `commands/capture.ts`)
- Comparison pipeline (`engine/comparison/` — schema check, LLM judge, embedding, variance)
- `commands/check.ts`, `commands/report.ts`, `commands/ideate.ts`
- Budget engine (`engine/budget.ts`)
- `generated_by` field from evals.json
- All associated tests

## What Stays (adapted)

- **Harness layer** (renamed from SkillAdapter) — `copilot-cli` adapter, extended to support with/without skill runs and session isolation
- **Inference layer** — copilot + github-models adapters with auto-resolution
- **Generator** (`engine/generator.ts`) — generates evals.json from SKILL.md (without assertions)
- **Config resolution** (`config.ts`) — defaults → project config → skill-dir config → CLI flags
- **Error hierarchy** (`errors.ts`)
- **Report layer** — terminal + HTML reporters, adapted for new data model

## Config

`snapeval.config.json`:

```json
{
  "harness": "copilot-cli",
  "inference": "auto",
  "workspace": "../{skill_name}-workspace",
  "runs": 1
}
```

- `harness`: which harness to use (default: `copilot-cli`)
- `inference`: which inference adapter for grading/generation (default: `auto`)
- `workspace`: workspace directory path template (default: sibling directory)
- `runs`: number of times to run each eval for statistical significance (default: 1)

## HTML Report (optional enhancement)

Interactive single-file HTML viewer showing:
- Per-eval assertion results (with/without skill side by side)
- Benchmark delta visualization
- Output diffs between with_skill and without_skill runs
- Pattern analysis flags (always-pass, always-fail, differentiating, outliers)
- Cross-iteration comparison when previous iterations exist

Not part of the agentskills.io spec but adds review value. Generated by `review` command.
