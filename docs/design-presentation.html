Why Plugin-First?

The V1 RCA showed that the skill IS the product. Users say "test my skill" to their AI agent. The AI reads the target skill, generates test cases, runs them, interprets results. The human never touches the CLI directly during interactive use.

Why Free Inference?

gpt-5-mini has a 0x multiplier on paid GitHub plans — completely free. In CI, GITHUB_TOKEN gives 150 req/day via GitHub Models API. Zero cost barrier to adoption.

02 Comparison Pipeline

Tiered comparison minimizes cost. Most snapshots are caught by free schema checks. Only ambiguous cases escalate to LLM judge.

Schema CheckFREE — structure/shape match
→
Pass?If identical structure, stop here
→
Embedding SimilarityCHEAP — cosine distance
→
Above threshold?If similarity > 0.85, pass
→
LLM JudgeEXPENSIVE — order-swap debiasing

Order-Swap Debiasing

The LLM judge runs twice: once with [baseline, new] order, once with [new, baseline]. Only agrees if both orderings produce the same verdict. Eliminates position bias that plagues single-pass LLM evaluation.

Variance Envelope

AI outputs are non-deterministic (58.8% of prompt+model combos drift on API updates). Baseline is captured N times to establish a variance envelope. New outputs are compared against the envelope, not a single snapshot.

03 Eval Format (agentskills.io conformant)

Follows the agentskills.io evaluation standard. AI generates evals.json; humans never author it manually.

my-skill/ ├── SKILL.md ← the skill under test ├── scripts/ ├── references/ └── evals/ ← snapeval workspace ├── evals.json ← AI-generated test cases ├── snapshots/ ← captured baseline outputs │ ├── scenario-1.snap.json │ ├── scenario-2.snap.json │ └── variance-envelope.json └── results/ └── iteration-1/ ├── grading.json ← per-assertion PASS/FAIL with evidence ├── timing.json ← tokens & duration └── benchmark.json ← aggregated pass_rate, cost, delta

04 Local User Story — Interactive Plugin

The developer never leaves their AI agent. The plugin handles everything.

● First-Time Setup — "Test my skill"
  1. Install the plugin
    gh copilot plugin install snapeval
  2. Ask the agent to evaluate a skill
    @snapeval evaluate my code-reviewer skill
  3. AI reads the target skill's SKILL.md and generates test scenarios
    snapeval I've read your code-reviewer skill. Here are 6 test scenarios: 1. Python file with SQL injection vulnerability 2. Clean TypeScript utility function (no issues expected) 3. Large 500-line file with mixed severity issues 4. File with only cosmetic/style issues 5. Empty file (edge case) 6. File in an unfamiliar language (Rust) Want to adjust these, or should I run them?
  4. User confirms. AI runs each scenario through Copilot CLI.
    snapeval Running 6 scenarios through Copilot CLI... ✓ Scenario 1 — captured (1.2s, 847 tokens) ✓ Scenario 2 — captured (0.9s, 612 tokens) ✓ Scenario 3 — captured (2.1s, 1,403 tokens) ✓ Scenario 4 — captured (1.0s, 721 tokens) ✓ Scenario 5 — captured (0.4s, 203 tokens) ✓ Scenario 6 — captured (1.5s, 994 tokens) Baseline captured. Total cost: $0.00 (Copilot gpt-5-mini) Saved to skills/code-reviewer/evals/
● Regression Check — "Did I break it?"
  1. Developer modifies the code-reviewer skill
    Changes the skill to skip files over 200 lines for performance
  2. Ask the agent to check for regressions
    @snapeval check my code-reviewer skill
  3. AI re-runs all scenarios and compares to baseline snapshots
    snapeval Running 6 scenarios against baseline... ✓ Scenario 1 — consistent (schema match) ✓ Scenario 2 — consistent (schema match) ✗ Scenario 3 — REGRESSION DETECTED Baseline: Full review of all 12 issues found Current: "File exceeds 200 lines, skipping detailed review" Similarity: 0.12 (threshold: 0.70) Tier: LLM Judge confirmed — behavioral change, not just wording ✓ Scenario 4 — consistent (embedding: 0.94) ✓ Scenario 5 — consistent (schema match) ✓ Scenario 6 — consistent (embedding: 0.88) 1 regression in 6 scenarios. Your skill now skips large files. This drops coverage for real-world codebases where large files often contain the most issues. Options: • Fix the skill to handle large files differently • @snapeval approve to accept this as new baseline

05 CI User Story — GitHub Actions

Automated regression detection on every PR. No human in the loop.

● Setup — Add workflow file
# .github/workflows/skill-eval.yml name: Skill Evaluation on: [pull_request] jobs: eval: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: snapeval/action@v1 with: skill-path: skills/code-reviewer adapter: copilot-cli # Uses GITHUB_TOKEN for inference — free # Reads existing evals/ directory for test cases & baselines # Posts results as PR check
● PR #42 — "Refactor code-reviewer for large files"
  1. Developer pushes PR with skill changes
    The PR modifies skills/code-reviewer/SKILL.md to skip files over 200 lines
  2. GitHub Action triggers snapeval check
    npx snapeval check --skill-path skills/code-reviewer --adapter copilot-cli --ci Loading baseline from evals/snapshots/... Running 6 scenarios... Comparing with tiered pipeline...
  3. Results posted as PR check
    ✗ snapeval — 1 regression detected ┌─────────────────────────────────────────────────┐ │ Skill Evaluation Report │ │ │ │ ✓ 5/6 scenarios passed │ │ ✗ 1 regression detected │ │ │ │ Scenario 3: "500-line file with mixed issues" │ │ Expected: Full review identifying 12 issues │ │ Got: "File too large, skipping" │ │ Similarity: 0.12 (threshold: 0.70) │ │ Detection: Tier 3 (LLM Judge confirmed) │ │ │ │ Cost: $0.00 (GitHub Models API) │ │ Duration: 8.2s │ │ │ │ Run npx snapeval approve to accept new baseline │ └─────────────────────────────────────────────────┘ Exit code: 1
  4. PR blocked — developer must fix or approve
    Developer either fixes the regression or runs npx snapeval approve --scenario 3 to accept the new behavior as baseline, then pushes the updated snapshot.

The Skeptic's Question, Answered

"Why not contribute to Promptfoo?" — Because Promptfoo is now OpenAI-owned, skill-level evaluation is architecturally different from prompt evaluation (skills have manifests, tool calls, multi-turn behavior, platform adapters), and the zero-assertion snapshot approach is a fundamentally different testing paradigm. snapeval doesn't compete with prompt eval tools — it creates a new category.
